Enhanced Statistical Test

Pattern Distribution Test

Quick Summary: The Pattern Distribution Test is a fast heuristic check that scans the bit sequence for three specific non-random structures: long runs of identical bits, near-perfect alternation, and immediately repeating 8-bit blocks. It is designed for short sequences where the formal NIST tests do not have enough data to be reliable.

What It Tests For

This test does not attempt a full statistical analysis of pattern frequencies. Instead, it performs three lightweight checks aimed at obvious giveaways that the input is not random:

  • Excessive run length: a single long sequence of identical bits
  • Near-perfect alternation: the input is almost entirely 010101... or 101010...
  • Repeating 8-bit blocks: two consecutive 8-bit blocks are byte-for-byte identical

The test passes when none of the three flags fire, and fails (with a list of issues) otherwise. There is no chi-square statistic and no p-value.

How It Works

Check 1: Maximum Consecutive Identical Bits

Scan the bit stream and record the length of the longest run of identical bits. The check fails if that run is longer than max(n / 4, 8) bits, where n is the total bit count:

n = 64 bits → flag if longest run > 16 bits n = 100 bits → flag if longest run > 25 bits n = 16 bits → flag if longest run > 8 bits (the floor)

The threshold scales with input size, so a long run is only flagged when it represents a sizeable fraction of the data.

Check 2: Alternating Ratio

Count the number of position pairs where consecutive bits differ. Divide by n to get the alternating ratio. The check fails if this ratio exceeds 0.9:

ratio = (number of i where bits[i] != bits[i-1]) / n Sequence 01010101... → ratio ≈ 1.0 → FAIL Sequence 11001010... → ratio ≈ 0.5 → pass Sequence 11111111... → ratio = 0.0 → pass (but Check 1 may flag it)

Check 3: Repeating 8-bit Block

Slide an 8-bit window across the sequence and check whether any 8-bit block is immediately followed by the exact same 8-bit block. The check is skipped for sequences shorter than 16 bits:

... 1011 0010 [10110010][10110010] 1100 ... ^^^^^^^^ ^^^^^^^^ two adjacent 8-bit blocks match → FAIL

Only adjacent duplicates are checked; non-adjacent recurrences of the same byte do not trigger the flag.

Result:
  • PASS: none of the three checks fired
  • FAIL: one or more checks fired; the failing check(s) are reported as issues

The reported "statistic" is simply the number of failing checks (0–3).

Why It Is a Heuristic

This test is intentionally narrow. It catches input that is obviously broken (e.g. a user pasting 0,1,0,1,0,1,... or 64 zero-bytes in a row), but it does not detect subtle non-randomness. A sequence that passes this test has only avoided the three specific failure modes — it has not been validated as random.

Important: Use this test as a fast sanity check. For real randomness assessment, run the full set of small-sequence and (where data permits) NIST SP 800-22 tests.

Common Failure Patterns

Example 1: Long Run of Identical Bits

Sequence: 11111111 11111111 11111111 00010110 n = 32, longest run of 1s = 24 Threshold = max(32/4, 8) = 8 24 > 8 → FAIL: "24 consecutive identical bits"

Example 2: Pure Alternation

Sequence: 01010101 01010101 01010101 01010101 Alternating ratio ≈ 31/32 ≈ 0.97 0.97 > 0.9 → FAIL: "97% alternating pattern"

Example 3: Repeating Byte

Sequence: ... 10110010 10110010 ... Two adjacent 8-bit windows are identical → FAIL: "Repeating block pattern detected"

Relationship to Other Tests

The three checks overlap with — but are far simpler than — the formal tests they resemble:

  • Longest Run Test (NIST 2.4): rigorous chi-square test of run-length distribution; this page's Check 1 is a single threshold
  • Runs Test (NIST 2.3): tests overall oscillation rate; this page's Check 2 only fires on near-perfect alternation
  • Frequency Test, Block Frequency, Serial: none of these block-frequency analyses are performed here

If sufficient data is available, prefer the formal tests over this heuristic.

When This Test Is Useful

  • Catching obviously degenerate input pasted by a user
  • Quick pre-screening before running heavier statistical tests
  • Short sequences where the formal NIST tests cannot run
